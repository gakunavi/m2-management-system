# 統合管理システム - プロジェクトPRD

## 1. プロジェクト概要

### 1.1 目的

会社全体および複数事業を横断して「顧客」「代理店」「営業案件」を一元管理するシステムを構築する。
既存のMOAG管理システムで培った業務ノウハウを活かしつつ、設計上の課題を解消し、事業の追加に耐えうる拡張性の高いアーキテクチャで1から構築する。

### 1.2 背景と課題

既存システム（MOAG管理システム）には以下の課題がある：

| 課題 | 詳細 |
|------|------|
| **単一事業にハードコード** | MOAG事業の概念（MO/MOAG/18ステップ）がコード全体に直接埋め込まれている |
| **共通コンポーネント化不足** | List画面・Detail画面・Form画面でロジックが重複している |
| **命名の不統一** | DB・API・フロントエンドでカラム名の新旧が混在している |
| **フロントエンドフックの分散** | useProjectData等のエンティティ固有フックが汎用化されていない |
| **レスポンス形式の不統一** | コントローラーごとにAPIレスポンス構造が微妙に異なる |

### 1.3 ゴール

- 複数事業を**設定ベース**で管理できるマルチ事業アーキテクチャ
- 顧客・代理店を**会社レベルで一元管理**し、事業間で共有
- 事業ごとに異なる営業フロー（ムーブメント）を**テンプレート定義**で対応
- 全社・事業別・代理店別の**数値可視化**（売上 + 案件パイプライン）
- 売上目標と実績の**予実管理**による業績可視化
- **厳格な命名規則**と**共通コンポーネント基盤**による保守性の高いコード

## 2. ユーザーと権限

### 2.1 ユーザー種別

| 種別 | 説明 | 権限レベル |
|------|------|-----------|
| **管理者（admin）** | システム管理者。全事業・全データへのフルアクセス | 全操作 |
| **担当者（staff）** | 所属事業の案件管理、顧客・代理店管理 | CRUD（所属事業） |
| **代理店管理者（partner_admin）** | 自社+下位代理店の関連事業データ閲覧・編集、代理店ユーザー管理 | CRUD（自社関連） |
| **代理店担当者（partner_staff）** | 自社+下位代理店の関連事業データ閲覧 | 閲覧のみ（自社関連） |

### 2.2 権限モデル

```
ユーザー ←→ 事業（多対多: user_business_assignments）
代理店  ←→ 事業（多対多: partner_business_links。事業内階層・手数料率も管理）
```

- 社内ユーザーは複数の事業に所属できる
- 代理店は複数の事業に関与できる
- データアクセスは所属事業でフィルタリングされる

**代理店の可視範囲:**
- 代理店ユーザーには自社が関与している事業と自社案件由来の集計を表示
- 下位代理店の数値も表示対象に含める
- 他代理店の数値と会社全体数値は表示しない

## 3. 機能要件

### 3.1 コア機能

#### F1: 事業管理
- 事業の登録・編集・無効化
- 事業ごとの営業ステータス定義
- 事業ごとのムーブメントテンプレート定義
- 事業固有フィールドの定義（JSON Schema）

#### F2: 顧客管理（会社レベル）
- 顧客の登録・編集・削除
- 顧客担当者の管理（1対多）
- 事業横断での顧客検索
- 顧客に紐づく全事業の案件一覧

#### F3: 代理店管理（会社レベル）
- 代理店の登録・編集・削除
- 代理店担当者の管理（1対多）
- 代理店と事業のリンク管理（事業内階層・手数料率含む）
- 代理店階層の2種類管理（全社マスタ階層 + 事業内階層）

#### F4: 案件管理（事業レベル）
- 事業に紐づく案件のCRUD
- 営業ステータス管理（事業ごとに定義可能）
- 案件番号の自動採番（事業プレフィックス付き）
- 事業固有フィールドの入力（動的フォーム）
- 1案件に紐づく代理店は1社のみ（単一代理店）

#### F5: ムーブメント管理（事業レベル）
- 事業ごとのステップテンプレート定義
- 案件のムーブメント進捗管理
- ステータス連動（設定ベース）
- ムーブメントログの記録

#### F6: ダッシュボード
- **全社ダッシュボード**: 全事業の売上合計、予実達成率、パイプライン概要
- **事業別ダッシュボード**: その事業の売上予実、案件ステータス分布、代理店別成績
- **代理店ビュー**: 自分の関与事業の売上実績、自分の案件リスト

#### F11: 予実管理（売上目標と実績）
- 事業別・月別の売上目標設定
- 代理店別・月別の売上目標設定（任意）
- 担当者別・月別の売上目標設定（任意）
- 目標に対する実績の自動集計（案件金額ベース）
- 達成率の可視化（ダッシュボードと連携）
- 目標未設定の期間は実績のみ表示

#### F12: 通知・アラート
- 案件ステータス変更時の通知（担当者・関係者へ）
- 受注予定日の超過アラート
- ムーブメント停滞アラート（一定期間進捗なし）
- システム通知（CSVインポート完了、エラー等）
- ヘッダーの通知ベルアイコン + 通知一覧ドロワー
- 既読/未読管理

### 3.2 補助機能

#### F7: CSV管理
- 事業テンプレートに基づく動的CSVインポート/エクスポート
- テンプレートダウンロード（事業ごと）
- バリデーション付きインポート

#### F8: ファイル管理
- 案件・顧客・代理店に紐づくファイルアップロード
- ファイルカテゴリ管理

#### F9: QA/問い合わせ管理
- QAカテゴリ・QA項目管理
- 問い合わせ管理

#### F10: 月次レポート
- 全社月次レポート
- 事業別月次レポート
- 代理店別月次レポート

#### F15: 一括操作（バッチアクション）
- 一覧画面でのチェックボックスによる複数行選択
- 選択行に対するステータス一括変更、担当者一括変更、一括削除
- 実行前の確認ダイアログ（破壊的操作は赤色表示）
- 操作に追加入力が必要な場合のインライン入力UI（変更先ステータス選択等）
- 権限に基づくアクション制御（例: 一括削除はadminのみ）

#### F16: 楽観的ロック（同時編集防止）
- 主要マスタ（顧客・代理店・案件）にバージョン番号を保持
- 更新時にバージョンチェックを行い、競合を検知
- 競合時は409 Conflictレスポンスと競合解決UI（最新データ再読み込み or 編集続行）
- Prisma middlewareによる自動バージョンチェック＋インクリメント

#### F17: 論理削除ポリシー
- 全削除操作は論理削除（is_active=false）で処理
- 無効レコードは一覧のデフォルト表示から除外（フィルターで表示可能）
- 無効レコードの詳細画面には「無効」バナーを表示
- 管理者による復元操作（is_active=trueに戻す）をサポート
- 子レコードは連鎖削除しない（紐づきを保持）

#### F13: データ品質管理
- 顧客・代理店の重複チェック（名称類似度による候補表示）
- 必須フィールドバリデーション（Zodスキーマ）
- 重複候補のマージ機能（将来拡張）

#### F14: 監査ログ
- 主要データ（顧客・代理店・案件・ムーブメント）の変更履歴を自動記録
- 変更前後の値を保持（フィールド単位のdiff）
- 操作者・操作日時・IPアドレスの記録
- 管理者向けの監査ログ閲覧画面

| 項目 | 要件 |

## 7. 設計原則

### 7.1 命名規則（厳格遵守）

#### データベース
- テーブル名: `snake_case`、複数形（`customers`, `partners`, `projects`）
- カラム名: `snake_case`、テーブルプレフィックス付与（`customer_name`, `partner_code`）
- 外部キー: `<参照先テーブル単数形>_id`（`customer_id`, `business_id`）
- タイムスタンプ: `created_at`, `updated_at`
- 操作者: `created_by`, `updated_by`

#### API
- エンドポイント: `/api/<version>/<resource>`（例: `/api/v1/customers`）
- レスポンスキー: `camelCase`（DBの`snake_case`から自動変換）
- レスポンス形式: 全エンドポイントで統一

```json
{
  "success": true,
  "data": { ... },
  "meta": {
    "total": 100,
    "page": 1,
    "pageSize": 20,
    "totalPages": 5
  }
}
```

エラー時:
```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "入力内容にエラーがあります",
    "details": [...]
  }
}
```

#### フロントエンド
- コンポーネント: `PascalCase`（`CustomerList.tsx`, `ProjectForm.tsx`）
- フック: `camelCase`、`use`プレフィックス（`useEntityList`, `useEntityForm`）
- 変数・関数: `camelCase`
- 定数: `UPPER_SNAKE_CASE`
- ファイル名: コンポーネントは`PascalCase.tsx`、それ以外は`camelCase.ts`

### 7.2 共通コンポーネント設計方針

**原則: Entity-Agnostic（エンティティ非依存）**

```
悪い例（現システムの課題）:
  useProjectData()     ← プロジェクト専用
  useProjectFilters()  ← プロジェクト専用
  useProjectTable()    ← プロジェクト専用

良い例（新システム）:
  useEntityList({ entityType, config })     ← 全エンティティ共通
  useEntityDetail({ entityType, id })       ← 全エンティティ共通
  useEntityForm({ entityType, schema })     ← 全エンティティ共通
```

各画面は**設定（config）を渡すだけ**で動作する。ロジックの重複は一切許容しない。

### 7.3 事業拡張性の原則

**「事業追加 = コード変更ゼロ」を目指す**

事業追加時に必要なのは：
1. `businesses`テーブルにレコード追加
2. `movement_templates`にステップ定義を追加
3. 営業ステータス定義を追加
4. 事業固有フィールドのJSON Schema定義を追加（案件・顧客それぞれ）
5. 売上計上ルールの設定（`business_config.revenueRecognition`）
6. 代理店の事業内階層・手数料率の設定

上記はすべてデータベース上の設定であり、コードの変更は不要。

### 7.4 事業横断マスタの拡張性

**顧客・代理店は事業横断のマスタだが、事業ごとの差異にも対応する**

| マスタ | 共通情報 | 事業固有情報 |
|-------|---------|------------|
| 顧客 | 会社名、住所、業種、反社チェック等 | `customer_business_links.link_custom_data`で事業ごとの追加情報を管理 |
| 顧客担当者 | 担当者名、部署、電話等 | `customer_contacts.business_id`で事業別の窓口担当を識別 |
| 代理店 | 会社名、全社マスタ階層（紹介親子）等 | `partner_business_links`で事業内階層・手数料率・事業固有情報を管理 |

**設計方針:**
- 顧客マスタ本体（`customers`）は全事業で共通の会社情報のみ持つ
- 事業固有の顧客情報は`customer_business_links.link_custom_data`（JSONB）に格納
- 顧客担当者は`business_id`で事業別に管理（NULLの場合は全事業共通）
- 案件フォームでの顧客担当者選択は、該当事業の担当者を優先表示し、なければ共通担当者をフォールバック
- 代理店階層は2種類: 全社マスタ階層（紹介親子）と事業内階層（報酬・表示用）は独立管理
- マスタ上の親子関係と事業内の1次店/2次店は一致しなくてよい

**採番ルール（確定）:**
- 顧客コード: `CST-0001` 形式（自動採番）
- 代理店コード: `AG-0001` 形式（自動採番。AGはagent由来）
- 案件番号: `{事業プレフィックス}-{顧客コード}-{連番}` 形式（例: `MG-CST0001-003`）

**重複チェック（確定）:**
- 顧客: 会社名 + 電話番号の組み合わせでNG
- 代理店: 同様の重複チェック方針

## 8. 既存システムから引き継ぐ仕様

詳細は `04_EXISTING_SPECS.md` を参照。

**データ移行: 不要** — 現行システムからのデータ移行は行わない。新システムは1から運用を開始する。

主な引き継ぎ項目：
- 営業ステータスの7段階（事業ごとにカスタマイズ可能に拡張）
- ムーブメントのステップ管理概念（18ステップを事業テンプレートとして一般化）
- ステータスとムーブメントの連動ロジック（設定ベースに一般化）
- CSV一括インポート/エクスポートの仕組み
- ファイル管理（UnifiedFileManager相当）
- ガントチャート表示
- 反社チェックステータス管理
