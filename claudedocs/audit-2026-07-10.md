### 追加で見つかった漏洩（監査後のポータル調査で発見・修正済み）

`GET /api/v1/portal/movements` が代理店に対して以下を返していた:
- `projectNotes`（社内メモ。案件一覧APIでは隠していたのに、ここでは無条件で返却。ポータルUIは使っていないため純粋な漏洩）
- `visibleToPartner=false` のフィールド値（`linkedFieldValue` 経由）。実機検証では **単価 900,000 / 500,000 円などが代理店に送信されていた**

いずれも修正し、ニーズ（`visibleToPartner=true`）の表示は維持されることを確認済み。

### A-4 の選択肢（要判断）

本番の `_prisma_migrations` は `scripts/baseline-migrations.sh` によって「全件適用済み」とマークされている一方、マイグレーションSQL自体には14テーブルのCREATEが存在しない。そのため:
- 本番へのデプロイは動く（新しいマイグレーションだけが実行される）
- **まっさらなDBでは `migrate deploy` が必ず失敗する**（実機で `relation "projects" does not exist` を確認）

修復には既存マイグレーションの整理が必要だが、やり方によっては本番の `migrate deploy` がチェックサム不一致で止まるリスクがある。本番DBは非公開サブネットにあり、こちらから `_prisma_migrations` の実際の中身を確認できないため、方針をユーザーと決めてから着手する。

### 本番反映の手順（A-1 / A-2）

**デプロイするだけでよい。手作業のスクリプト実行は不要。**

`docker-entrypoint.sh` が起動時に次の順で処理する:
1. `prisma migrate deploy` — カラムを `user_password_plain` → `user_password_encrypted` にリネーム（この時点の中身はまだ平文。アプリは平文フォールバックで正常表示する）
2. `node ./scripts/encrypt-existing-passwords.js` — 残っている平文を暗号化。冪等なのでコンテナが再起動して何度走っても安全

`ENCRYPTION_KEY` は未設定でよい（`NEXTAUTH_SECRET` から鍵が派生される）。ただし **`NEXTAUTH_SECRET` を今後変更すると発行パスワードが復号できなくなる**点に注意。

全レコードの暗号化完了後、`src/lib/password-vault.ts` の平文フォールバック分岐を削除してよい。

#### スクリプトの安全装置（実機で検証済み）

このスクリプトは事故を起こしやすい条件が揃っているため、以下を実装している:

- **接続先DBを必ず表示してから実行する。** `@prisma/client` は cwd に関係なくプロジェクトの `.env` を自動読み込みするため、環境変数を指定し忘れると意図しないDBに無言で接続する（実際に検証中、開発機の `.env` の `DATABASE_URL` を拾って別プロジェクトのDBを指しかけた）
- **手元のターミナルから実行したときは `yes` の確認を求める**（`--yes` で省略可）。ECS の entrypoint は TTY が無いのでそのまま進む
- **既存の暗号文を現在の鍵で復号できなければ、DBを一切書き換えずに中止する。** 異なる鍵で暗号化した行が混在すると、そのユーザーのパスワードは永久に読めなくなる
- **鍵が未設定なのに平文が残っている場合は、静かに素通りせず失敗する**
- 書き込む前に、暗号化した値が元の平文に復号できることを確認する
- entrypoint はこのスクリプトの失敗ではコンテナを落とさない（アプリは平文フォールバックで動くため）。ログに WARNING が出る

手動で実行する場合:
```
DATABASE_URL=... NEXTAUTH_SECRET=... npm run db:encrypt-passwords
```

## A. 重大な不具合（早急な対応を推奨）

### A-1. 代理店ユーザーへの広範な情報漏洩（IDOR / 認可漏れ）【最重要】

middleware は `/api/v1/*` を「セッション有無」でしか守っておらず、ロールチェックは各ルート任せ。書き込み系（POST/PATCH/DELETE）はほぼ全て `['admin','staff']` チェックがあるが、**読み取り系（GET）が系統的に抜けている**。代理店ユーザー（partner_admin / partner_staff）が正規ログインのまま、ブラウザのfetchで以下へ直接アクセスできる:

| エンドポイント | 漏れる情報 |
|---|---|
| `GET /api/v1/customers`・`/customers/[id]` | 全顧客のPII・社内メモ・連絡先 |
| `GET /api/v1/partners`・`/partners/[id]` | 他代理店（競合含む）の階層・取引情報 |
| `GET /customers/[id]/bank-accounts`・`/partners/[id]/bank-accounts` | **銀行口座番号・名義人（無マスク）** |
| `GET /customers/[id]/contacts`・`/partners/[id]/contacts` | 担当者氏名・電話・メール・名刺画像URL |
| `GET /partners/[id]/business-links` | **手数料率（commissionRate）**・事業別カスタムデータ |
| `GET /customers/csv`・`/partners/csv` | 全件一括CSVダウンロード（POSTインポート側だけ保護されている） |
| `GET /projects/[id]/files`・`/projects/[id]/movements` | 任意案件のファイル直リンク・ムーブメント |
| `GET /customers/duplicate-check`・`/partners/duplicate-check` | 名前部分一致で電話番号付き情報（存在確認オラクル） |
| group-tree / candidates / filter-options 系 | 全代理店の名称・コード・階層構造 |

さらに書き込みIDORが1件: `PATCH /projects/[id]/movements/[movementId]` はロールチェックが一切なく、代理店ユーザーが任意案件のムーブメント状態や `linkedFieldUpdate` 経由で projectCustomData を書き換え可能。

**スコープバイパス（else-if アンチパターン）**: `projects/route.ts:116-136` と `projects/csv/route.ts:61-79` は `if (businessIdParam) {...} else if (partner系スコープ) {...}` の構造のため、`?businessId=X` を付けるだけで代理店スコープと `portalVisible` 制約が外れる。staff もアサイン外事業を閲覧可能。加えて `if (user.partnerId)` のフェイルオープン（partnerId未設定の代理店ユーザーはスコープ無し全件閲覧）。

**修正方針（構造的に一括対応）**:
1. `withApiAuth` にロール要件を渡せるようにする: `withApiAuth(handler, { roles: ['admin','staff'] })` を導入し、GET含む全ハンドラーに明示適用
2. または middleware で代理店ロールの社内API（/api/v1/customers, /partners, /projects の非ポータル用途）を403で一元遮断
3. スコープ適用はパラメータ分岐（businessId等）と**独立**に必ずANDで適用。partner ロールで partnerId 未設定は403（フェイルクローズ）

※ ポータル系API（portal/*）・notifications・partner-staff・ai/*・users・tasks は適切に保護されていることを確認済み。

### A-2. パスワードの平文保存 + API返却

- `prisma/schema.prisma:15` に `user_password_plain VARCHAR(100)` があり、bcryptハッシュと並存
- `users/route.ts:52,186`・`partner-staff/route.ts:63,179` が作成時に平文を保存し、**GET応答で平文を返却**している

DB流出・バックアップ流出・画面閲覧で全アカウントの生パスワードが漏れる。パスワード使い回しがあれば外部サービスにも波及。

**修正方針**: カラム削除マイグレーション + API応答から除去。「発行パスワードの通知」が必要なら発行直後のみ一時表示（保存しない）。既存ユーザーにはパスワード変更を案内。

### A-3. リマインダー・タスク通知のcronが本番で一度も動いていない（可能性大）

- `/api/v1/cron/reminders`（期日リマインダーメール）と `/api/v1/cron/task-notifications` は実装済み・CRON_SECRET保護済み
- しかし **AWS側に呼び出し元が存在しない**（EventBridge Rules/Scheduler 共に空、GitHub Actions にも cron なし）
- Secrets Manager に `m2/cron-secret` が用意されているのに使われていない

**修正方針**: EventBridge Scheduler で 1日1回（例: 朝9時JST）ALB経由でPOSTを叩くスケジュールを作成（Lambda または API Destination + Bearer認証）。夜間停止運用（後述C案）を採る場合は稼働時間内に設定。

### A-4. マイグレーション履歴が破綻（新環境が構築不能）

schema.prisma にある14テーブル（projects, partner_business_links, notifications 等）の CREATE TABLE がどのマイグレーションにも存在しない。`db push` か手動SQLで本番適用された模様。**マイグレーションから新環境（ステージング・災害復旧・開発DB）を構築すると途中で必ず失敗する**。Dockerfile起動時の `migrate deploy` は既存本番DBだから通っているだけ。

**修正方針**: `prisma migrate diff --from-migrations --to-schema-datamodel` で差分SQLを生成しベースライン補完、または履歴スカッシュ + `migrate resolve --applied`。以後 `db push` 禁止を運用ルール化。

### A-5. 顧客・代理店の詳細画面で「事業カスタム情報」が常に空

`useEntityDetail` が `?businessId=` を付けずにGETするため、API は `linkCustomData: {}` を返し、詳細画面の事業別カスタムフィールドは常に「-」表示。編集フォームとクロスエンティティタブは businessId 付きで取得しており、**詳細画面だけ欠落**（CLAUDE.mdのREAD詳細経路チェック違反の再発）。

**修正方針**: `useEntityDetail` に businessId を渡し queryKey にも含める。

### A-6. インライン編集のID衝突バグ（案件行が顧客データで破壊される）

`use-inline-cell-edit.ts:66` の同一エンティティ判定が `updated.id === row.id` のみ。案件一覧から顧客フィールドを編集した際、偶然「案件ID == 顧客ID」だと（auto-incrementなので普通に起きる）、案件行が顧客オブジェクトで丸ごと置換され表示が破損。しかもその場合キャッシュ無効化もスキップされる。

**修正方針**: `customPatch.endpoint` のパスと entityType の対応で判定する。

## B. 中程度の問題

### 整合性・楽観的ロック
1. **projects DELETE に version increment なし**（customers/partners はあり）。restore も同様（projects のみ欠落）。batch論理削除3種も version 未更新
2. **子・中間テーブルに楽観ロックなし**: PartnerBusinessLink（手数料率・階層を持つ）等は last-write-wins。特に business-links/[linkId] PATCH は 409 判定なし、かつ existing 読み取りがトランザクション外（TOCTOU）
3. **business-links/toggle が階層継承を呼ばない**: 一覧のトグルで作ったリンクは事業別階層が継承されない（POST側は f91e761 で対応済みなのに非対称）
4. **convert-to-qa の check-then-act がトランザクション外**: 同時実行でQA二重作成
5. **inquiry 削除時にS3ファイルが残る**（DB は Cascade、ストレージは放置。qa/items は正しく削除している）
6. **partner batch 論理削除で BP申込書を物理削除**: restore してもファイルは戻らない
7. **documents PUT のファイル差し替え順序**: 旧削除→新アップロードの順なので、失敗すると旧ファイル消失

### ストレージ・セキュリティ
8. **`public/uploads` が無認証配信**: 名刺画像・問い合わせ添付・事業ドキュメントがURL既知なら認証なしで取得可能（推測困難なキーのみが防壁）。→ S3非公開バケット + 署名付きURL、またはアクセス制御付きプロキシAPIへ
9. **upload DELETE にパストラバーサル**: `key.join('/')` に `..` 除去がなく、LocalStorageAdapter で basePath 外のファイル削除が可能（本番S3では影響低いが要修正）
10. **MIME検証不備**: inquiries 添付は type 未検証。汎用 upload はクライアント申告の file.type のみ + 拡張子は元ファイル名由来 → `.html`/`.svg` を配信パスに置けて保存型XSSの余地
11. **レートリミット**: X-Forwarded-For 最左値使用でヘッダ詐称により回避可能 + インメモリのためインスタンス間非共有
12. **CSPヘッダー未設定**（他のセキュリティヘッダーはあり）

### パフォーマンス（データ増加で劣化）
13. **カスタム列ソート/フィルタ時に全件取得→メモリ内ソート**（projects/customers/partners 共通）。件数増で線形悪化
14. **通知ポップオーバーが60秒間隔でフルリストをポーリング**（全ページ・全タブ常駐）。軽量な `useUnreadCount` は実装済みなのに未使用の死にコード。→ たまにしか使わないのに Fargate/RDS を常に起こしているのはこれも一因
15. **group-tree のBFSがノード毎に2〜3クエリ**（最大300クエリ）→ 一括取得か再帰CTEへ
16. **CSVインポートが行毎に逐次クエリの120秒トランザクション** → プリフェッチ + createMany へ
17. **inquiries / qa 一覧がページネーションなしの全件取得**
18. **インデックス不足**: CustomerBusinessLink に businessId 単独インデックスなし（Partner側にはある）。FK インデックス欠落多数（Task.createdById, Inquiry.inquiryProjectId 等）

### フロントエンド
19. **customer/partner フォームが動的必須フィールドをバリデーションしない**（project は buildDynamicFieldSchema で対応済み、非対称）
20. **customers/partners 一覧が config ロード完了を待たず描画**: その窓でインライン編集すると businessId なし PATCH になりデータが消えて見える
21. **動的カスタム列の楽観的更新が画面に反映されない**（フラットキーのみ更新、render はネスト参照）
22. ポータルダッシュボード・問い合わせカテゴリ取得にエラー表示なし（無言で空表示）
23. インライン編集の非APIエラーが無通知（赤枠のみ）
24. ビュー保存上限判定の不整合（共有ビュー含む/含まないが混在）

### データモデル
25. **PartnerBusinessLink.businessParentId の整合性がDB保証されない**（親が同事業にリンク済みかは制約なし）→ 複合FK追加を推奨
26. **階層再計算が既存の階層番号を振り直す**（帳票に載せた番号が変わり得る）+ 再帰に深さ上限なし（循環データで無限再帰）
27. strategy-report の親解決フォールバックで事業別1次店が誤判定され得る
28. ステータス系が全てVarChar（enum/CHECKなし）、SavedTableView の同名重複可、TaskAssignee のNULL unique 非対象

### 案3: 構成変更で最安化（-約¥8,000〜9,500/月）
